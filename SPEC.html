Live Claude Coach — Specification

A self-extending habit coach for Claude Code usage · v0.1 · 2026-07-21 · status: stages 1–3 implemented 2026-07-21 (P2 & P8 implemented 2026-07-22; live system in ~/.claude/coach/, git-tracked)

1. Background & goal

An audit of 458 main sessions (plus 886 subagent/workflow transcripts, ~1 GB) found that a handful of usage habits dominate cost: 12 sessions account for ~49% of 13.25 billion lifetime cache-read tokens. The measured habits:

  • God sessions — one giant context used as project memory (132 sessions peaked >150k tokens; worst: 1.29B cache-read tokens over 26 days).
  • Micro-prompt babysitting — "commit and push" ×51, "please continue" ×12, each forcing a full-context pass in a fat session.
  • Invisible automation fleet — 202/458 sessions are a per-commit security-review hook, all on frontier models.
  • Startup toll — ~53k-token baseline context loaded into every session by hooks/plugins/skills.
  • Small tool frictions — 4,382 cd-prefixed Bash calls, ~700 cat/find/grep-in-Bash, 871 redundant re-reads, 178 >8k-char pastes.

The goal is a tutor that (a) catches known habits at the moment they recur, (b) discovers new habits as they form, and (c) surfaces unknown unknowns — including habits present since day one, capabilities never used, and patterns no current metric measures. The user's memory is explicitly not part of the design: every cue lives in the environment.

Core architectural principle — the compilation ladder. Expensive intelligence (LLM analysis) runs rarely, on small aggregates; its discoveries are compiled downward into zero-token deterministic sensors (hooks, scripts, statusline) that run always. The enforcement layer never costs tokens when behavior is good, never hallucinates, and never depends on a model being loaded. The discovery layer is the only part that spends tokens — a bounded ~30–60k/week on a cheap model, versus the 13.2B historical burn it polices (<0.001%/yr at current rates).

All tutor state lives in one directory: ~/.claude/coach/ — rules.json, state.json, snapshots/ (weekly stats), COACH-MODEL.md, reports/. Everything is plain text, human-editable, and portable (consistent with the user's MMS portable-memory philosophy).

3. The pieces

P1 — Nudge hook (fast loop)build 1st

Motivation

Catches the two habits worth ~half of historical spend (god sessions, babysitting prompts) at the exact moment of recurrence. Removes memory from the loop entirely: the cue is environmental, fires at the point of behavior, and escalating counters make cost visible ("babysitting prompt #7 this week"). This is the enforcement floor everything else compiles into.

Implementation

  • One UserPromptSubmit hook in ~/.claude/settings.json running tutor-hook.py (stdlib Python, <150 lines).
  • Hook receives JSON on stdin (prompt, transcript_path, session_id); emits additionalContext with a one-line nudge, or nothing.
  • Rules are data, not code: rules.json holds an array of rules — {id, kind: regex|threshold, pattern/metric, threshold, message, cooldown, status}. The slow loop appends here; no code changes to evolve.
  • Seed rules: (1) babysitting regex commit (and )?push|please continue|^continue$|what('s| is) (the )?next; (2) god-session threshold — tail-read the last ~50 transcript lines, take the last assistant usage block, context = input + cache_read + cache_creation; nudge at 150k, escalate at 300k with a concrete instruction ("run handoff-after-clear, then /clear"); (3) fat-paste warning at >8k-char prompt.
  • state.json keeps per-rule counters + last-fired timestamps for cooldowns (no nagging every turn) and feeds the weekly report.

Cost & benefits

0 tokens when no rule fires (script runs outside the model, milliseconds); ~40 tokens per nudge. Deterministic — cannot hallucinate, cannot drift, works identically on any model. Rule file is human-readable and diffable.

P2 — Statusline ambient cue (ambient loop)built 2026-07-22

Motivation

"Make it obvious" — a permanently visible context meter and habit grade (🟢/🟡/🔴) means drift is seen before the P1 nudge fires, with zero interruption. Set aside from the first build because P1 delivers most of the value and the statusline is pure polish — but specified here so it isn't forgotten.

Implementation

  • statusLine command in settings pointing at a script that reads the same tail-of-transcript context estimate as P1 plus state.json nudge counters.
  • Renders e.g. ctx 142k 🟡 · nudges this wk: 3. The statusline-setup agent already installed can do the wiring.

Cost & benefits

0 tokens, ever — the statusline runs client-side. Reuses P1's code; ~30 lines of glue.

P3 — Weekly report card (slow loop, reporting half)build 2nd

Motivation

"Make it satisfying." Habits stick when progress is visible. Trend lines ("babysitting 51 → 9, sessions over 150k: 6 → 1, ~2.1B tokens saved") are the reward loop. Also the human approval gate: every rule/sensor the tutor wants to add is listed here for a yes/no — the tutor never modifies itself silently.

Implementation

  • Weekly job (piggyback on the existing 17:30 cron, or Windows Task Scheduler / claude scheduled agent) runs tutor-stats.py — the audit script from 2026-07-21, extended — over transcripts modified in the last 7 days.
  • Writes snapshots/YYYY-WW.json; renders an HTML report card into reports/ with week-over-week deltas, rule-fire counts from state.json, and pending proposals.
  • Metric set is itself data-driven where possible (a metrics manifest), so the meta loop can add sensors without rewriting the script core.

Cost & benefits

0 tokens for stats + report generation (pure script). Streams JSONL; never loads transcripts into a model context.

P4 — Sensor A: presence anomalies (slow loop)build 2nd

Motivation

New bad habits are, by definition, new repetition — and repetition always surfaces in habit-agnostic distributions (top repeated short prompts, first-prompt clusters, context percentiles, tool mix, interrupt rate, model mix, re-read counts, paste sizes). This is exactly how the original audit found "commit and push" without knowing to look for it. Catches habits you develop next month, not just today's list.

Implementation

  • tutor-stats.py computes the distributions; a diff step compares this week's snapshot to trailing 4-week baseline and emits only significant deltas (new prompt n-gram clusters, percentile shifts, tool-mix spikes) — a few KB.
  • One headless call: claude -p --model claude-haiku-4-5 "<standing prompt> + <delta JSON>" asking: what new repeated pattern is emerging, is it wasteful, and if so propose a detector rule in rules.json schema.
  • Proposals land in the P3 report card for approval; on yes, appended to rules.json → enforcement compiles down to P1 at zero marginal cost.

Cost & benefits

~5k tokens/week on Haiku. Sees only aggregates, never raw transcripts — cheap and privacy-tight by construction.

P5 — Sensor B: absence anomalies (slow loop)build 2nd

Motivation

Deltas can never catch what you never do — absence has no frequency. Yet non-use was half the audit's findings: handoff-after-clear installed but invoked once; Haiku used in 89 of 50,000 messages; /compact 8 times ever; 101 skills of unknown utilization. Also covers the platform moving underneath you: a new Claude Code feature can silently turn a current pattern into a legacy workaround.

Implementation

  • Inventory script enumerates capabilities: ~/.claude/skills/, plugin lists, hooks, model tiers, native features (compact, worktrees, background tasks, subagents) → cross-references invocation counts already extracted by tutor-stats.py.
  • Emits an "owned vs used" table with a staleness score; the weekly LLM call gets it alongside Sensor A's deltas and answers: "what did not-using X cost this week?"
  • Monthly: WebFetch the Claude Code changelog/release notes; LLM checks new features against the current habit model ("does anything here obsolete a pattern in COACH-MODEL.md?").
  • Side benefit: the same table drives plugin/skill pruning suggestions (the ~53k startup toll from the audit).

Cost & benefits

~3k tokens/week (shares the Sensor A call); changelog check ~2k/month. Inventory itself is 0 tokens.

P6 — Sensor C: out-of-model discovery (slow loop)build 3rd

Motivation

Sensors A and B are trapped inside the metric set — if the stats script doesn't compute it, it doesn't exist. The only escape from your own feature space is reading raw behavior cold. This is the sensor that surfaces true unknown unknowns, including habits present since day one (flat lines that no delta will ever flag). It is the single part of the system that makes it "living" rather than a dashboard, and the only recurring part that spends real tokens.

Implementation

  • Weekly stratified sample of 3 sessions: one uniformly random, one most-expensive-this-week, one most-recent.
  • Extractor compresses each to a behavioral trace (~10k chars): all user turns verbatim + first 200 chars of each assistant reply + tool-call names — behavior, not content.
  • One headless call (Sonnet — this task rewards judgment) with the standing question: "What is this person doing inefficiently that NONE of the following current metrics would catch? <metric manifest>. Propose either a rules.json detector or a new metric for the stats script."
  • Findings feed the P3 approval gate; approved metrics get added to the manifest → Sensor A watches them from then on.

Cost & benefits

~20–50k tokens/week on Sonnet. Bounded by the extractor cap regardless of session size. Discovery compiles downward: each finding becomes a 0-token sensor.

P7 — Meta loop: COACH-MODEL.md + self-extensionbuild 3rd

Motivation

A living system needs memory of itself and pressure to grow. Without rule retirement, the tutor fossilizes into a museum of 2026 habits and nags about behaviors long fixed; without the "unasked question" step, its senses never expand. The model file tracks who you are now, not who you were in July.

Implementation

  • COACH-MODEL.md — one human-readable file: known habits (status: active / improving / retired), current sensor manifest, open hypotheses, decision log.
  • Every weekly run's LLM call ends with two mandatory outputs: (1) "what question should this audit have asked that it didn't?" → proposed sensor; (2) rules with zero fires in 30 days → proposed retirement.
  • All mutations flow through the P3 approval gate. The file is git-tracked in the tutor directory so the tutor's evolution is itself auditable.

Cost & benefits

~2k tokens/week (part of the weekly call). Keeps rules.json small — retirement is what keeps the fast loop fast and the nudges credible.

P8 — Quarterly blind-spot passbuilt 2026-07-22

Motivation

The weekly loop normalizes its own blind spots — whatever it consistently overlooks becomes invisible to it. A quarterly third-party audit of the tutor itself, using the already-installed blind-spot-pass skill with COACH-MODEL.md as input ("my Claude Code usage — what does this model of me not see?"), resets the frame. Set aside: valuable, but only after the weekly loop has a quarter of history to critique.

Implementation

  • Calendar reminder or quarterly cron prompting a manual run: /blind-spot-pass + the model file + latest 4 report cards.
  • Output reviewed by hand; accepted findings become sensors/rules via the normal gate.

Cost & benefits

~50k tokens/quarter. Human-in-the-loop by design.

P9 — Adjacent one-time fixes (not tutor components)one-time

Motivation

Audit findings that need a single repair, not ongoing coaching. Listed so they're not lost; excluded from the tutor because a tutor for a solved problem is dead weight.

  • Gate the security-review hook — skip doc/lockfile-only diffs, batch per-push, pin to Haiku/Sonnet. Halves session count permanently (202/458 sessions).
  • Plugin/skill pruning — via plugin-skill-auditor/cco; demote rarely-used globals to project scope to cut the ~53k startup toll. Sensor B keeps this honest afterward.
  • Permission allowlist — run fewer-permission-prompts; removes the incentive behind 4,382 cd-prefixed compound Bash calls.
  • CLAUDE.md standing rules — "commit and push after each module without asking" in active project CLAUDE.md files; converts 51 typed prompts into zero.

Cost & benefits

0 recurring tokens; each is minutes of setup with permanent payback.

Context for scale: ~60k tokens/week ≈ 3M/year, against a measured historical burn of 13.25B — the tutor costs <0.03% of what it polices, and the enforcement layer that runs thousands of times per week costs nothing at all.

Benefits beyond token cost: deterministic enforcement can't hallucinate or drift; nudges arrive at the moment of behavior (no reliance on memory — the original requirement); every rule and model file is plain text, human-editable, git-diffable, and portable across providers; the approval gate means the tutor never rewires itself silently; and each discovery permanently upgrades the free layer, so the system gets cheaper per insight over time.
